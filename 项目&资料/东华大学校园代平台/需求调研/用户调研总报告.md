# 用户调研总报告

## 一、调研背景与目标

### 1.1 调研背景

我在校园里做过 4 周的代服务（代取快递、代签到、代领资料），发现同学的刚需分散在 3 类场景：

- 代服务：代取快递、跨校区代送、**非核心课程代签到 / 代记笔记**（用户常吐槽 “赶课来不及取快递”“选修课临时有事需要代签”）；
- 二手交易：教材 / 资料卖不出去、买错版本；
- 新生指南：找不到快递站、流程看不懂。

目前这些需求没有校园专属平台整合，且代上课类服务存在 “合规边界模糊” 的问题，因此我通过调研明确需求、划分合规场景，设计贴合东华学生的服务平台。

### 1.2 调研目标

1. 覆盖**代业务（含代上课合规场景）、二手交易、新生指南**三大模块，量化各场景的需求频次与痛点；
2. 明确代上课业务的**合规边界**（仅支持非核心课程代签、代领资料等，排除违规场景），设计风险控制方案；
3. 输出全模块 “痛点 - 需求 - 功能” 映射关系，划分需求优先级；
4. 形成可落地的产品规划，支撑后续原型、PRD 与实习简历展示。

## 四、核心调研发现（全业务模块）

### 4.1 代业务模块（含代上课）：高频刚需，痛点集中

#### 4.1.1 需求频次与合规场景（代上课）

- 数据：72 位代业务用户中，

  44.4%（32 人）有代上课相关需求

  ，其中：

  - 代领资料（78.1%）> 代签到（65.6%）> 代记笔记（43.8%）；
  - 每月需求 1-2 次占 65.6%，3-4 次占 28.1%。

- 访谈案例：

  - 延安路大三男生：“跨校区上选修课，每月 2 次赶不上，需要代签 + 代领课件”；
  - 松江考研女生：“非核心课要复习，需要代签 + 记考点笔记”。

#### 4.1.2 全模块核心痛点

| 痛点类型       | 问卷占比 | 对应场景            | 访谈案例佐证                                               |
| -------------- | -------- | ------------------- | ---------------------------------------------------------- |
| 服务不靠谱     | 42.5%    | 代取 / 代签         | 大二女生 “代取快递被弄混”；考研女生 “代签迟到被记缺勤”     |
| 沟通低效       | 35%      | 跨校区代送 / 代上课 | 大三男生 “没说清教室，代签者跑错楼”；男生仅 55% 能说清需求 |
| 价格不透明     | 27.5%    | 跨校区代送 / 代上课 | 大二男生 “代签有人要 10 元，有人要 20 元”                  |
| 信息传递不完整 | 22%      | 代上课（代记笔记）  | 考研女生 “代记笔记漏考点，影响复习”                        |

### 4.2 二手交易模块：精准需求，信任缺失

- 数据：65% 用户需要 “专业 + 年级标签”，45% 男生遇过 “货不对板” 纠纷；
- 访谈案例：
  - 大三女生：“挂教材 3 天 5 人问版本，最后送学妹”；
  - 大四男生：“买二手耳机有划痕，没退款”。

### 4.3 新生指南模块：刚需集中，信息零散

- 数据：81.25% 新生需要 “流程指引”，75% 需要 “校园导航”；
- 访谈案例：
  - 大一男生：“找快递站绕 20 分钟”；
  - 大一女生：“办医保问 3 人才弄明白”。

## 五、核心用户画像（全业务覆盖）

### 5.1 高效付费型（大二大三女生）

- 标签：代取高频、付费意愿强；
- 行为：每周代取 4 次，愿花 8 元加急；
- 需求：靠谱服务、梯度加急。

### 5.2 价格敏感型（大二大三男生）

- 标签：跨校区代送 / 代签、沟通低效；
- 行为：代签最多花 12 元，常漏说地址；
- 需求：固定定价、标准化模板。

### 5.3 考研刚需型（大四女生）

- 标签：代记笔记 / 代买饭、资料交易；
- 行为：天天泡图书馆，愿花 25 元代签 + 记笔记；
- 需求：定时服务、资料版本保真。

### 5.4 新生适应型（大一学生）

- 标签：流程 / 导航刚需、低付费；
- 行为：找快递站绕路，买教材怕错版；
- 需求：分步指引、精准导航。

## 六、全业务需求清单与优先级

### 6.1 P0 级（必须有，上线核心）

| 需求 ID | 模块     | 需求描述                                                     | 对应痛点             |
| ------- | -------- | ------------------------------------------------------------ | -------------------- |
| D01     | 代业务   | 代上课合规场景入口（代签 / 代领 / 代记）+ 服务凭证上传（签到照 / 笔记照） | 信任风险、信息不完整 |
| D02     | 代业务   | 需求发布标准化模板（代取填取件码 / 代签填课程信息）          | 沟通低效             |
| D03     | 二手交易 | 专业 + 年级标签（发布强制填，搜索支持筛选）                  | 信息不精准           |
| D04     | 二手交易 | 交易担保（先付平台，确认后转卖家）+ 瑕疵标注                 | 交易纠纷             |
| D05     | 新生指南 | 校园精准导航（标快递站）+ 流程分步指引（报到 / 医保）        | 导航不准、流程零散   |

### 6.2 P1 级（应该有，迭代优化）

| 需求 ID | 模块     | 需求描述                                  | 对应痛点   |
| ------- | -------- | ----------------------------------------- | ---------- |
| D06     | 代业务   | 代上课固定定价（代签 10 元 / 代记 15 元） | 价格不透明 |
| D07     | 代业务   | 接单者信誉标签（无差错达人）              | 服务不靠谱 |
| D08     | 二手交易 | 自动回复模板（卖家设版本 / 自提点）       | 沟通反复   |
| D09     | 新生指南 | 学长学姐答疑入口（匿名提问）              | 答疑无门   |
| D10     | 代业务   | 课程提醒（代签前 1 小时推送）             | 签到迟到   |

### 6.3 P2 级（可以有，资源充足时）

| 需求 ID | 模块     | 需求描述                            | 对应痛点     |
| ------- | -------- | ----------------------------------- | ------------ |
| D11     | 代业务   | 代记笔记标准化模板（填考点 / 作业） | 信息不完整   |
| D12     | 二手交易 | 考研资料价格参考（平台标区间）      | 价格虚高     |

## 七、风险评估与应对策略

| 风险类型       | 对应模块 | 应对策略                                                     |
| -------------- | -------- | ------------------------------------------------------------ |
| 代上课违规风险 | 代业务   | 1. 注册签《合规协议》，拦截 “核心课 / 考试” 关键词；2. 人工审需求，违规则封禁 |
| 交易资金风险   | 二手交易 | 用校园卡虚拟余额，避免现金流转；设 24 小时确认期             |
| 信息传递纠纷   | 代上课   | 明确验收标准（笔记含 3 个考点）；纠纷以凭证为依据            |

## 八、后续工作规划（全业务落地）

### 8.1 阶段 1：核心功能落地（1-2 周）

- 原型：用墨刀做 P0 级功能（代上课合规入口、二手标签、新生导航）；
- PRD：明确各功能规则（如代上课凭证要求）；
- 评审：邀 5 位同学测原型，优化流程。

### 8.2 阶段 2：迭代优化（2-3 周）

- 完善 P1 级功能（代上课定价、信誉标签）；
- 设计 UI（校园清新风格）；
- 招募 20 位用户做可用性测试。

### 8.3 阶段 3：灰度测试（3-4 周）

- 低代码开发 MVP 版（P0 功能）；
- 双校区各招 50 人测试，跟踪使用率；
- 收集反馈，迭代优化。
